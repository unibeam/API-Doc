MFA Reality — What Current Methods Actually Protect Against

Most enterprise MFA deployments add a second factor that is still, fundamentally, a string the attacker can obtain without touching the user's device. Understanding why requires distinguishing what a method claims to protect from what it actually protects.

The shared weakness across OTP, SMS, and push

TOTP codes, SMS one-time passwords, and push approvals all reduce to a signal that can be intercepted, ported, or socially engineered — without any access to the hardware the user holds. This keeps them effectively in the "something you know" risk category:

The gap most MFA programs leave open

Two password layers is not MFA. If both factors can be obtained remotely without physical access to the user's hardware, the authentication boundary is the same as password-only — the second factor adds friction, not a meaningful security boundary.

What "True Possession" Requires

The operative security property of "something you have" is that the credential cannot leave the hardware. A factor that can be copied, exported, or redirected provides possession-like friction — not a possession boundary.

SIM hardware as a secure element

A SIM card is a smart card running inside a tamper-resistant hardware module, separate from the device's main processor and OS. The Unibeam Applet is installed inside this environment — it has access to the SIM's secure storage but cannot be read by the host OS, any application, or any external tool. An attacker with full control of the Android or iOS device cannot extract the SIM's credentials.

The operational advantage over hardware tokens

External hardware tokens provide strong hardware-bound security but require physical distribution, replacement on loss, and create gap periods where a weaker fallback must be used. Unibeam delivers equivalent hardware security via the SIM already in the user's phone, provisioned remotely with no distribution overhead and no gap periods.

How Unibeam Authentication Works

Registration vs. ongoing sign-in — an important distinction

Unibeam has two distinct phases. Registration is a one-time setup step where the employee confirms their identity. Every subsequent sign-in is fully passive — no codes, no approvals, no user action. Any evaluation of Unibeam's security properties should hold both phases to the same standard.

⚡ Registration — once only

The employee goes to the security portal, selects Unibeam Auth, and confirms their identity with a short number-match challenge on their phone. This establishes the SIM-to-identity binding. It happens once per device/SIM combination.

✅ Every sign-in after that

Unibeam silently verifies the SIM in the background. No code. No push notification. No user action. The employee signs in normally — Unibeam's challenge runs without interrupting the sign-in flow.

Authentication flow (post-registration)

#ActorWhat happens
1UserSubmits primary credential (password) to Entra.
2EntraValidates password and delegates the second factor to Unibeam.
3UnibeamIdentifies the user by their registered mobile number and email address, then initiates a silent SIM authentication challenge.
4SIMThe Unibeam Applet on the SIM performs a challenge-response with the employee to grant a validated authentication. No user interaction. No code. No push notification.
5UnibeamReceives confirmation from the Applet and issues a signed identity confirmation to Entra including SIMId, DeviceId, and ChangeTimestamp.
6EntraVerifies the confirmation against Unibeam's service and grants the session.

Why there is nothing to intercept

The challenge in step 4 occurs between the Unibeam Applet on the SIM and Unibeam's service, over a channel not accessible from the public internet. There is no code generated that the user sees, types, or transmits — so there is nothing for an adversary-in-the-middle to capture. The subscriber's credentials never leave the SIM hardware.

SIM hardware vs. phone number

Authentication is bound to the credentials secured within the physical SIM hardware — not the phone number. A SIM swap assigns a number to a different SIM; because the credentials live on the original SIM hardware, the attacker's SIM will fail the authentication challenge regardless of the number it holds.

Identity Signals

Each successful authentication event produces three unique identifiers that Unibeam writes to the user's account in Entra via the Microsoft Graph API. These provide your security team with persistent, queryable signals for detection and response.

SIMId
SIM hardware identity

A unique, persistent identifier representing the specific SIM hardware used in the authentication. Internally derived — not based on any readable SIM attribute such as ICCID. Changes only when the physical SIM changes.

DeviceId
Device hardware identity

A unique identifier representing the physical device seen at authentication time. Independent of MDM or Intune enrolment. Stable for a given device — changes only when the user authenticates from a different physical device.

ChangeTimestamp
Identity change signal

A timestamp set whenever a new SIMId or DeviceId is observed for a user — indicating that the SIM hardware, the device hardware, or both changed since the last authentication.

⚠️ Implementation note — for your security architect

Unibeam writes SIMId, DeviceId, and ChangeTimestamp to the user object in Entra as custom security attributes via Microsoft Graph. Your security team can query these values through the Graph API and build detection rules in your SIEM or security tooling. Enforcement policies based on these values — such as blocking sign-ins when ChangeTimestamp is recent — require configuration in your environment (e.g. Conditional Access rules referencing the attributes, or SIEM-driven response workflows). These are not automatically enforced by Entra without that configuration. Confirm the specific enforcement mechanism with your Unibeam integration team before rollout.

Trust Model & Integration

What you are trusting

When Entra receives and verifies Unibeam's authentication response, it trusts Unibeam's service as a certified External Authentication Method. The challenge itself is handled entirely by the Unibeam Applet on the SIM, which owns the identifier-based challenge end-to-end.

Scope of access granted to Unibeam

Access typeWhatWhy
ReadUser's registered mobile number and email addressTo identify and route the SIM authentication challenge
WriteSIMId, DeviceId, ChangeTimestamp as custom security attributesTo record identity signals on the user object after each authentication
No accessPasswords, group membership, email content, calendar, files, or any other directory data—

Permissions are granted via one-time admin consent and are scoped to the UnibeamAuth attribute set. The service principal cannot read or write outside that scope.

Response verification

Entra verifies every authentication response from Unibeam before granting a session. Responses that were not issued by Unibeam for the specific sign-in request will be rejected.

Availability dependency

Unibeam's platform availability is a dependency for MFA. If Unibeam is unreachable, the second factor fails. Define a fallback policy in Conditional Access before rollout — for example, routing to a backup method or requiring step-up approval for users in the targeted group when Unibeam is unavailable.

Coverage requirements

SIM authentication requires the employee's device to have an active SIM at authentication time. Wi-Fi-only devices (tablets without a SIM) and devices in no-signal environments cannot complete the authentication challenge. These cases need a defined fallback policy before deployment.

Summary for the CISO

Unibeam replaces a phishable second factor with a hardware-bound one, eliminates the operational overhead of external token distribution, and produces per-authentication identity signals your security team can act on. The one-time registration involves a number-match confirmation; every subsequent sign-in is passive. The safety-delay and SIM-change detection capabilities depend on your team configuring enforcement rules against the attributes Unibeam writes — they are signals, not automatic blocks. Confirm implementation details with your Unibeam contact before committing to specific detection workflows.
