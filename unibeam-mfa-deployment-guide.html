MFA Factor Framework

Microsoft's authentication security model classifies second factors into three categories. Understanding where each MFA method genuinely sits in this model — not just where vendors claim it sits — is the prerequisite for evaluating risk.

Why category label is not enough

Reason

TOTP and SMS OTP are commonly classified as "something you have" because they require a phone. This is misleading. The security property of "something you have" is that the secret cannot leave the hardware. For TOTP, the seed is a string that can be exported, backed up, and phished. For SMS, the phone number (MSISDN) is a logical identifier that can be ported to another SIM by a carrier support call. Neither meets the actual security criterion of the category.

The table below shows where each method genuinely sits:

PasswordKnowPhishable, leakable, guessable

The design criterion

A genuine second factor must not be reducible to a string the attacker can obtain remotely. If the secret can be intercepted or ported, the factor provides one layer of security, not two. TOTP and SMS OTP both reduce to strings. FIDO2 and SIM authentication do not.

1
The Threat Model

The problem Unibeam addresses is not weak passwords — it is that the second factor in most enterprise MFA deployments is also phishable.

Why TOTP fails

Reason

TOTP codes are a shared secret between the user's device and the IdP. An adversary-in-the-middle (AiTM) proxy captures the code in real time as the user submits it, and replays it within the 30-second validity window. The code has no binding to the device, the network, or the session — it is a string. AiTM kits targeting TOTP are commodity tooling.

Why SMS / WhatsApp OTP fails

Reason

OTP delivery over SMS has two structural weaknesses. First, SS7 — the 1970s-era global signalling protocol — allows a party with network access to redirect messages to any MSISDN. Second, SIM swap: a social-engineering attack against carrier support staff that ports the victim's number to an attacker-controlled SIM. Both attacks require no access to the user's device. WhatsApp OTP uses the same underlying MSISDN as SMS — a successful SIM swap compromises both channels simultaneously.

Why hardware tokens are not a universal answer

Reason

FIDO2 tokens are phishing-resistant. The operational constraint is that they require procurement, distribution, replacement on loss, and physical presence to enrol. At organisations with thousands of distributed, remote, or shift-based employees, this creates provisioning gaps — periods where users are either locked out or are being authenticated by a weaker fallback. Gaps in coverage are exploitable.

The Attack Surface Unibeam Reduces

AiTM phishing, SS7 interception, and SIM swap all require the attacker to operate against a signal (code, message) that is separate from the identity hardware. Unibeam moves authentication entirely to the SIM — a piece of hardware the attacker cannot access remotely.

How Authentication Works

Unibeam integrates with Microsoft Entra ID as a certified External Authentication Method over OIDC. The following describes what actually happens on the network during an MFA event.

Protocol flow

#ActorWhat happens
1UserSubmits primary credential (password) to Entra.
2EntraValidates password, delegates MFA to the registered External Authentication Method (Unibeam) via OIDC redirect.
3UnibeamReads the user's mobilePhone attribute from Entra. Initiates a silent SIM authentication challenge directly with the mobile operator's network via the SIM Application Toolkit (SIM AT) channel.
4SIM / CarrierThe SIM applet on the device performs a challenge-response with the carrier network, secured by the subscriber's known credentials. No user interaction. No code. No push notification.
5UnibeamReceives confirmation from the carrier. Issues a signed OIDC id_token to Entra with claims including SIMId, DeviceId, and ChangeTimestamp.
6EntraValidates the token signature against Unibeam's JWKS endpoint, grants the session.

Why there is nothing to intercept

Reason

The challenge-response in step 4 occurs on the carrier's internal network between the network authentication centre (AuC) and the SIM applet. This channel is not accessible from the public internet. There is no code generated that the user sees, types, or transmits — so there is nothing for a proxy or SS7 attacker to capture. The subscriber's known credentials never leave the SIM.

SIM hardware vs. phone number

Reason

Authentication is bound to the subscriber's credentials secured within the physical SIM hardware — not to the phone number (MSISDN). A SIM swap assigns a number to a different SIM; because the subscriber's credentials live on the original SIM hardware, a swapped number does not inherit the authentication capability. The attacker's SIM will fail the carrier challenge.

Identity Signals

Each successful authentication event produces three claims in the OIDC token, stored as Entra Custom Security Attributes. These are not metadata — they are signals your detection and response stack can act on.

SIMId — SIM hardware identity

Why it matters

Unibeam generates a unique identifier that represents the SIM hardware that completed the carrier challenge. This identifier is not derived from the ICCID or any externally readable SIM property — it is Unibeam's own stable reference to that specific SIM. It remains constant as long as the same SIM is in use. When SIMId changes between two authentications for the same user, the SIM hardware itself changed — either an expected swap (new plan, new phone) or an unexpected one that warrants investigation.

DeviceId — Device hardware identity

Why it matters

Unibeam generates a unique identifier that represents the physical device hardware seen at authentication time. This identifier is independent of any MDM or Intune enrolment — no device management agent is required. It remains stable for a given device and changes only when the user authenticates from a different physical device. Use it to enforce one-user-one-device policies and detect account sharing (same user principal, multiple concurrent DeviceId values).

ChangeTimestamp — Identity change signal

Why it matters

A timestamp that is set whenever a new SIMId or DeviceId is observed for a user — meaning the SIM hardware, the device hardware, or both, changed since the last authentication. It does not depend exclusively on any single carrier-side identifier. This is the basis for a configurable safety delay.

The safety delay

A common post-breach pattern: an attacker obtains valid credentials, acquires a new device, and attempts immediate sign-in. The ChangeTimestamp exposes this — a new identity signal will have just been recorded:

IF now() − ChangeTimestamp < threshold THEN block login OR restrict access to sensitive areas

The threshold is configurable per group. A legitimate employee who just got a new phone will also trigger this — the difference is they can be granted an exemption by IT or complete a step-up challenge. An attacker with stolen credentials and a new device cannot.

ScenarioSignalResponse
Normal sign-in No change to SIMId or DeviceId Allow
Employee changed phone New DeviceId, ChangeTimestamp recent Step-up or IT exemption
Credential theft + new device New DeviceId or new SIMId, ChangeTimestamp just set Block + alert
SIM swap at carrier New SIMId, carrier challenge fails on original SIM Fails auth + SIEM event

Trust Model & Integration

What you are trusting

Reason

When Entra validates Unibeam's id_token, it is trusting Unibeam's OIDC endpoint and its signing keys (fetched from the discovery endpoint). Unibeam, in turn, relies on the carrier's network connectivity to reach the SIM — but the challenge itself is handled entirely by the Unibeam Applet running on the SIM. The carrier is an enabler, not the root of trust: it provides the transport and the SIM as a secure execution environment, while the Unibeam Applet owns the cryptographic challenge-response. You are not adding a new root of trust; you are leveraging existing network infrastructure to surface a security layer that Unibeam controls end-to-end.

What Unibeam can read from your tenant

ReadUser's mobilePhone attribute — to route the SIM challenge
WriteSIMId, DeviceId, ChangeTimestamp as Custom Security Attributes on the user object
No accessPasswords, group membership, email, calendar, files, or any other directory data

Permissions are granted once via admin consent and are scoped to the UnibeamAuth attribute set. The service principal cannot read or write outside that scope.

Token validation

Reason

Entra validates the id_token signature against Unibeam's published signing keys via the OIDC discovery endpoint. Tokens issued for a different audience are rejected by Entra. A forged token without a valid signature from Unibeam's private key will not pass validation.

Dependency to understand

Unibeam's platform availability is a dependency for MFA. If Unibeam is unreachable, the second factor fails. Entra fallback behaviour (e.g. allowing password-only or routing to a backup method) should be defined in your Conditional Access policy before rollout.

Operational Considerations

Coverage requirements

SIM authentication requires the employee's device to have a carrier signal at authentication time. Wi-Fi-only devices (tablets without a SIM) or devices in no-signal environments cannot complete the carrier challenge. These cases need a defined fallback policy.

Employee onboarding

Each user registers once via mysignins.microsoft.com/security-info. Registration links the user's Entra identity to their SIM. After that, authentication is silent — no ongoing user action required.

Enrolment scope

Unibeam Auth is targeted per group in Entra. Users outside targeted groups are unaffected. This allows staged rollout: start with a pilot group, validate the full flow end-to-end, then expand. The safety delay and SIMId change signals are only populated after first enrolment.

SIEM integration

The three custom attributes (SIMId, DeviceId, ChangeTimestamp) are written to the Entra user object on each auth event and are queryable via Microsoft Graph. Any SIEM with Entra sign-in log ingestion will receive these values in the token claims. Build detection rules around SIMId rotation and ChangeTimestamp recency using your existing tooling — no separate Unibeam SIEM connector is required.
