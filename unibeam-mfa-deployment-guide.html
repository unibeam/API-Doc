Unibeam External MFA

Unibeam Auth integrates with Microsoft Entra ID as a certified External Authentication Method (EAM) over OIDC. It replaces the second factor in your existing Conditional Access policies with SIM-based authentication — leveraging the cryptographic security of the subscriber's SIM hardware rather than a code, push notification, or application secret.

What "External Authentication Method" means

Microsoft Entra EAM is a standards-based extension point that allows a certified third-party identity provider to serve as the MFA step in an Entra sign-in flow. Unibeam plugs into this slot — Entra handles the primary credential, Unibeam handles the second factor, and the result is a signed OIDC token returned to Entra.

The sections below explain what the current MFA landscape looks like, why existing methods leave security gaps, and why Unibeam's architecture closes those gaps at the hardware level.

1.a  MFA Today — TOTP, OTP & Microsoft Authenticator

The three dominant MFA methods in enterprise Microsoft environments each address account takeover in theory — but each has a structural weakness that keeps them effectively within the "something you know" threat category, despite being labelled as a second factor.

TOTP

Time-based one-time passwords. A shared seed generates a 6-digit code every 30 seconds.

Seed is a string — exportable, backable, phishable
Codes captured in real time by AiTM proxy
30-second replay window; commodity tooling available
Does not require physical device — only the seed
SMS / WhatsApp OTP

One-time codes delivered over the carrier network to a phone number.

SS7 vulnerabilities allow interception without device access
SIM swap ports the number to an attacker SIM via carrier support
WhatsApp OTP shares the same MSISDN — SIM swap compromises both channels
Relies on number as identity, not hardware
Microsoft Authenticator

Push notifications or number-matching via Microsoft's authenticator app.

Number matching reduces accidental approvals (MFA fatigue)
App secret extractable from device backup or rooted device
Push fatigue / social engineering remains viable at scale
Security depends on OS-level device integrity — no dedicated hardware

Where each method actually sits in the security model

The operative security property of "something you have" is that the credential cannot leave the hardware. TOTP, SMS OTP, and push-based apps all fail this test:

The design criterion

A genuine second factor must not be reducible to a string the attacker can obtain remotely. If the secret can be intercepted, ported, or extracted from a backup, the factor provides one layer of security — not two. TOTP, SMS OTP, and app-based authenticators all reduce to strings. FIDO2 and SIM authentication do not.

1.b  Why Unibeam is Different from "Old-Generation Authenticators"

Legacy authenticators — TOTP apps, SMS OTP, and push-based solutions — share a common architecture: they authenticate a phone number or an application secret stored in software. Unibeam takes a fundamentally different approach: authentication is bound to a cryptographic identity secured inside the SIM hardware itself, verified by the mobile carrier's network in real time.

The core architectural difference

Why it matters

Old-generation authenticators generate or receive a code and ask the user to type or approve it. That code is the authentication proof — and because it is a string, it can be captured. Unibeam generates no code. The proof of identity is a cryptographic challenge-response between the Unibeam Applet on the SIM and the carrier's Authentication Centre (AuC), taking place entirely within the carrier network. The user sees nothing; there is nothing for an attacker to intercept.

Why "hardware-backed app" is still a software solution

Some solutions market themselves as "hardware-backed" because they use a phone's secure enclave to store an app secret. A phone's secure enclave protects against local extraction — it does not protect against the application being enrolled on a backup device, or the phone number being ported. Unibeam's security boundary is the SIM hardware and the carrier network, neither of which is reachable from the public internet.

1.b.1  Unibeam as an Application + Hardware Solution — SIM & eSIM Security vs. Apps and External Hardware Tokens (Yubico)

Unibeam combines two layers that no pure-software or pure-hardware solution provides together: a carrier-integrated application layer (the Unibeam Applet) running inside certified, tamper-resistant hardware (the SIM or eSIM). Understanding how this compares to software authenticators and to dedicated external hardware tokens like Yubico YubiKeys clarifies where each approach's security boundary actually sits.

SIM and eSIM as a secure element

Why it matters

A SIM card is a Java Card-compliant smart card running inside a certified tamper-resistant hardware module. It meets the same hardware security standards (ISO/IEC 7816, GlobalPlatform) as dedicated external tokens. The Unibeam Applet is installed inside this environment — it has access to the SIM's cryptographic primitives and its own isolated storage, but cannot be read by the host OS, the application layer, or any external tool. An attacker with full control of the Android or iOS device cannot extract credentials from the SIM.

eSIM (eUICC) brings the same hardware security model with the addition of remote provisioning. The applet is deployed and updated over-the-air through the carrier's SM-DP+ server — eliminating the physical distribution requirement that makes hardware tokens operationally difficult at scale. For enterprises with large, distributed, or remote workforces, eSIM means coverage can be provisioned the same day it is needed, with no gap period where a weaker fallback method is in use.

The combined advantage

Unibeam delivers the hardware-security guarantees of a FIDO2 token — non-exportable credentials, no phishable code — while removing the operational constraints that prevent universal deployment. How Authentication Works

Unibeam integrates with Microsoft Entra ID as a certified External Authentication Method over OIDC. The following describes what actually happens on the network during an MFA event.

Protocol flow

#ActorWhat happens
1UserSubmits primary credential (password) to Entra.
2EntraValidates password, delegates MFA to the registered External Authentication Method (Unibeam) via OIDC redirect.
3UnibeamReads the user's mobilePhone attribute from Entra. Initiates a silent SIM authentication challenge directly with the mobile operator's network via the SIM Application Toolkit (SIM AT) channel.
4SIM / CarrierThe Unibeam Applet on the SIM performs a challenge-response with the carrier's Authentication Centre (AuC), secured by the subscriber's hardware-bound credentials. No user interaction. No code. No push notification.
5UnibeamReceives confirmation from the carrier. Issues a signed OIDC id_token to Entra with claims including SIMId, DeviceId, and ChangeTimestamp.
6EntraValidates the token signature against Unibeam's JWKS endpoint, grants the session.

Why there is nothing to intercept

Reason

The challenge-response in step 4 occurs on the carrier's internal network between the AuC and the Unibeam Applet on the SIM. This channel is not accessible from the public internet. There is no code generated that the user sees, types, or transmits — so there is nothing for a proxy or SS7 attacker to capture. The subscriber's credentials never leave the SIM.

SIM hardware vs. phone number

Reason

Authentication is bound to the credentials secured within the physical SIM hardware — not to the phone number (MSISDN). A SIM swap assigns a number to a different SIM; because the credentials live on the original SIM hardware, a swapped number does not inherit the authentication capability. The attacker's SIM will fail the carrier challenge.

3
Identity Signals

Each successful authentication event produces three claims in the OIDC token, stored as Entra Custom Security Attributes. These are not metadata — they are signals your detection and response stack can act on.

SIMId — SIM hardware identity

Why it matters

Unibeam generates a unique identifier representing the SIM hardware that completed the carrier challenge. It is not derived from the ICCID or any externally readable SIM property. It remains constant as long as the same SIM is in use. When SIMId changes between two authentications for the same user, the SIM hardware itself changed — either an expected swap or an unexpected one that warrants investigation.

DeviceId — Device hardware identity

Why it matters

A unique identifier representing the physical device hardware seen at authentication time. Independent of MDM or Intune enrolment — no device management agent is required. It remains stable for a given device and changes only when the user authenticates from a different physical device. Use it to enforce one-user-one-device policies and detect account sharing.

ChangeTimestamp — Identity change signal

Why it matters

A timestamp set whenever a new SIMId or DeviceId is observed for a user. This is the basis for a configurable safety delay:

IF now() − ChangeTimestamp < threshold THEN block login OR restrict access to sensitive areas

A legitimate employee who just got a new phone will also trigger this — the difference is they can be granted an exemption by IT. An attacker with stolen credentials and a new device cannot.

ScenarioSignalResponse
Normal sign-in No change to SIMId or DeviceId Allow
Employee changed phone New DeviceId, ChangeTimestamp recent Step-up or IT exemption
Credential theft + new device New DeviceId or new SIMId, ChangeTimestamp just set Block + alert
SIM swap at carrier New SIMId, carrier challenge fails on original SIM Fails auth + SIEM event

Trust Model & Integration

What you are trusting

Reason

When Entra validates Unibeam's id_token, it trusts Unibeam's OIDC endpoint and signing keys. Unibeam relies on carrier network connectivity to reach the SIM, but the challenge itself is handled entirely by the Unibeam Applet on the SIM. The carrier is an enabler, not the root of trust — it provides the transport and the SIM as a secure execution environment, while the Unibeam Applet owns the cryptographic challenge-response end-to-end.

What Unibeam can read from your tenant

ReadUser's mobilePhone attribute — to route the SIM challenge
WriteSIMId, DeviceId, ChangeTimestamp as Custom Security Attributes on the user object
No accessPasswords, group membership, email, calendar, files, or any other directory data

Permissions are granted once via admin consent and are scoped to the UnibeamAuth attribute set. The service principal cannot read or write outside that scope.

Token validation

Reason

Entra validates the id_token signature against Unibeam's published signing keys via the OIDC discovery endpoint. Tokens issued for a different audience are rejected. A forged token without a valid signature from Unibeam's private key will not pass validation.

Dependency to understand

Unibeam's platform availability is a dependency for MFA. If Unibeam is unreachable, the second factor fails. Entra fallback behaviour (e.g. allowing password-only or routing to a backup method) should be defined in your Conditional Access policy before rollout.

Operational Considerations

Coverage requirements

SIM authentication requires the employee's device to have a carrier signal at authentication time. Wi-Fi-only devices (tablets without a SIM) or devices in no-signal environments cannot complete the carrier challenge. These cases need a defined fallback policy.

Employee onboarding

Each user registers once via mysignins.microsoft.com/security-info. Registration links the user's Entra identity to their SIM. After that, authentication is silent — no ongoing user action required.

Enrolment scope

Unibeam Auth is targeted per group in Entra. Users outside targeted groups are unaffected. This allows staged rollout: start with a pilot group, validate the full flow end-to-end, then expand. The safety delay and SIMId change signals are only populated after first enrolment.

SIEM integration

The three custom attributes (SIMId, DeviceId, ChangeTimestamp) are written to the Entra user object on each auth event and are queryable via Microsoft Graph. Any SIEM with Entra sign-in log ingestion receives these values in the token claims. Build detection rules around SIMId rotation and ChangeTimestamp recency using your existing tooling — no separate Unibeam SIEM connector is required.
